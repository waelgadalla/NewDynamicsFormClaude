?? DynamicForms.Core.V2 Design Analysis

Part 2: Weaknesses, Limitations, and Missing Features

? Design Weaknesses (CONS)

CRITICAL 1. Limited Field Type Support

  • Issue: Only 4 type-specific configurations exist:
    • TextAreaConfig
    • FileUploadConfig
    • DateRangeConfig
    • ModalTableConfig
  • Missing Common Types:
    • NumberConfig (min/max, step, precision)
    • CurrencyConfig (currency code, formatting)
    • RichTextConfig (allowed tags, max length)
    • SignatureConfig (format, dimensions)
    • AddressConfig (country-specific formats)
    • PhoneConfig (country code, format validation)
    • RangeSliderConfig (min/max/step)
    • ColorPickerConfig
    • RatingConfig (max rating, icons)
  • Impact: HIGH - Limits real-world usability

CRITICAL 2. No Cascading Dropdown Implementation

  • Issue: RelationshipType.Cascade enum exists but no implementation
  • No way to filter child dropdown options based on parent selection
  • Common use case: Country ? Province/State ? City
  • Impact: HIGH - Frequently requested feature
// Currently defined but not implemented
public enum RelationshipType {
    Container = 0,
    Conditional = 1,
    Cascade = 2,      // ?? No implementation
    Validation = 3
}

WARNING 3. Conditional Logic Limited

  • Current: ConditionalRule exists but only basic operations
  • Missing:
    • Complex expressions (AND/OR combinations)
    • Comparison operators beyond equals (greater than, less than, between, contains)
    • Date-based conditions (age calculations, date ranges)
    • Multi-field conditions (if field A = X AND field B = Y)
  • Impact: MEDIUM - Limits complex form logic

WARNING 4. No Field Calculation/Formula Support

  • No way to define calculated fields (e.g., total = quantity * price)
  • No expression evaluation engine
  • Forces business logic into client-side code
  • Use Cases: Tax calculations, subtotals, date differences, scoring
  • Impact: MEDIUM - Common requirement in business forms

WARNING 5. Workflow Branching Incomplete

  • ConditionalBranch exists in schema but no runtime execution
  • No workflow state management
  • No support for:
    • Parallel paths (approval workflows)
    • Loop-back to previous steps
    • Dynamic step generation
  • Impact: MEDIUM - Limits workflow complexity

WARNING 6. No Built-in Persistence Implementation

  • IFormModuleRepository is just an interface
  • No reference implementation (SQL Server, SQLite, MongoDB)
  • Developers must implement from scratch
  • Impact: MEDIUM - Increases integration effort

WARNING 7. Limited Validation Features

  • Missing Validations:
    • Credit card validation
    • URL/URI validation
    • Postal code/ZIP code validation
    • IP address validation
    • Custom async validators with remote API calls
    • Cross-field dependencies (end date > start date)
  • No validation error severity levels (error vs warning vs info)
  • Impact: LOW - Can be added via custom rules

WARNING 8. No Field-Level Permissions/Security

  • No support for role-based field visibility
  • No read-only vs editable based on user
  • No field-level encryption hints
  • Use Case: Admin sees salary field, employee doesn't
  • Impact: MEDIUM - Security concern for sensitive data

WARNING 9. No Versioning/Change Tracking Built-in

  • Schema has Version property but no version management logic
  • No schema migration utilities
  • No change history or audit trail
  • Impact: LOW - Can be handled externally

WARNING 10. No Default Value Expressions

  • Fields have no default value support
  • No support for dynamic defaults (e.g., "today's date", "current user")
  • Forces initialization logic into client code
  • Impact: LOW - Workaround is easy

?? Potential Issues & Edge Cases

1. Performance with Large Forms

  • Scenario: Form with 500+ fields
  • Concern: Hierarchy building may be slow
  • Mitigation: Caching, lazy loading, pagination

2. Circular Reference Detection

  • Current: Detection exists but auto-fix may drop fields unexpectedly
  • Concern: Silent data loss if not logged properly
  • Recommendation: Add strict mode that throws exceptions

3. ExtendedProperties Type Safety

  • JsonElement? ExtendedProperties allows arbitrary JSON
  • No compile-time safety
  • Could lead to runtime deserialization errors
  • Trade-off: Flexibility vs type safety

4. Multilingual Limitations

  • Only English/French supported
  • Adding new languages requires schema changes
  • Better approach: Resource key + external localization service

5. No Event System

  • No events for field changes, validation, navigation
  • Makes it harder to build reactive UIs
  • No hooks for logging/auditing
