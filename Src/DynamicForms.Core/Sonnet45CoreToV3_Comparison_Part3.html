Implementation Checklist

  • ? Create FieldTemplateSchema record in Schemas folder
  • ? Create IFieldTemplateProvider interface in Services folder
  • ? Implement InMemoryFieldTemplateProvider
  • ? Add 5-10 built-in templates (Contact, Address, Organization, Financial, Emergency)
  • ? Register in ServiceCollectionExtensions

PORT 2. Draft System

  • ? Create DraftSchema record in Schemas folder
  • ? Create IDraftRepository interface in Services folder
  • ? Implement InMemoryDraftRepository
  • ? Add auto-expiration logic (24-hour default)
  • ? Create DraftService for business logic
  • ? Register in ServiceCollectionExtensions
  • ? Add unit tests
  • ? (Optional) Create EF Core implementation in separate Data project

ENHANCE 3. Advanced Conditional Rules

  • ? Create AdvancedConditionalRule record
  • ? Add LogicalOperator enum (And, Or, Not)
  • ? Create ConditionExpression record
  • ? Implement condition evaluator service
  • ? Update FormFieldSchema to support new rules
  • ? Add unit tests for complex expressions

ENHANCE 4. Advanced Visibility Controls

  • ? Add IsVisibleInEditor property to FormFieldSchema
  • ? Add IsVisibleInDisplay property
  • ? Add IsActive property (field active in system)
  • ? Update hierarchy service to respect visibility flags
  • ? Add unit tests

SKIP 5. Species Autocomplete (Domain-Specific)

  • ? Too specialized for core library
  • ? Better as separate plugin/extension

SKIP 6. Direct EF Core Entities in V3

  • ? Violates V3's immutability principle
  • ? Breaks clean architecture separation
  • ? Use separate Data project for EF entities instead

?? What NOT to Port

Features to Avoid

  • ? Mutable Entity Classes
    • Reason: V3 is built on immutability for thread safety
    • Solution: Keep immutable records, use separate EF entities in Data project
  • ? TextResource / FieldDatabase Nested Objects
    • Reason: V3's flat structure is simpler and more serializable
    • Solution: Keep V3's inline properties (LabelEn, LabelFr, ColumnName, etc.)
  • ? Species Autocomplete
    • Reason: Too domain-specific for core library
    • Solution: Create as separate extension package if needed
  • ? Direct EF Core Navigation Properties
    • Reason: Breaks clean architecture and serialization
    • Solution: Use runtime hierarchy building instead
